# File-first workflow runs (runner/worker model)

This document describes the **file-first** workflow run storage format used by the ClawRecipes workflow runner.

Goals:
- deterministic + auditable (everything important is written to disk)
- replay/debug-friendly (inspect `run.json` + node outputs)
- works without a DB

## Run folder layout

A workflow run is stored under:

```
shared-context/
  workflow-runs/
    <runId>/
      run.json
      node-outputs/
        001-<nodeId>.json
        002-<nodeId>.json
      artifacts/
        ... optional files written by tools / nodes
      approvals/
        approval.json
    <runId>.run.json   # convenience alias (same content as <runId>/run.json)
```

Notes:
- `runId` is a timestamp-ish id generated by the CLI.
- `run.json` is the canonical source of truth for run + node state.
- `node-outputs/*.json` are **JSON-first** outputs for every executed node.
- `artifacts/` is optional and may include markdown, binary blobs, tool payloads, etc.
- `approvals/approval.json` exists only for approval-gated runs.

## `run.json` (high level)

`run.json` records:
- workflow metadata (teamId, workflowId/file, createdAt)
- overall run status (`queued`, `running`, `awaiting_approval`, `waiting_workers`, `completed`, `failed`, ...)
- per-node state (status + timestamps)
- an append-only event log (what happened, when)

If you need to understand *why* something happened, start here.

## Node outputs

Each executed node must produce a JSON output file:

```
shared-context/workflow-runs/<runId>/node-outputs/###-<nodeId>.json
```

This output is intended to be stable and machine-readable.

## Approval gating

Approval is a first-class state:
- When an approval node is reached, the run transitions to `awaiting_approval`.
- The runner writes `approvals/approval.json` (decision record).
- Once recorded, the run can be resumed.

## Pull-based execution (runner vs workers)

The system is split into two roles:

1) **Runner** (scheduler)
- claims queued runs
- computes next runnable nodes based on the workflow graph (edges + conditions)
- enqueues per-node execution tasks into per-agent queues

2) **Workers** (executors)
- each agent runs `worker-tick` to pull tasks from its own queue
- executes the node and writes JSON output + updates `run.json`

## CLI quickstart

### Enqueue / run a workflow

Workflows are read from:

```
shared-context/workflows/
```

Run once (manual trigger):

```bash
openclaw recipes workflows run --team-id <teamId> --workflow-file <file.workflow.json>
```

### Runner tick (cron-friendly)

Process queued runs (scheduler):

```bash
openclaw recipes workflows runner-tick --team-id <teamId>
# or
openclaw recipes workflows runner-once --team-id <teamId>
```

### Worker tick (per agent)

Execute queued node tasks for a specific agent:

```bash
openclaw recipes workflows worker-tick --team-id <teamId> --agent-id <agentId>
```

### Approval flows

Record an approval decision:

```bash
openclaw recipes workflows approve --team-id <teamId> --run-id <runId> --decision approve
# decision: approve | reject
```

Resume after a decision is recorded:

```bash
openclaw recipes workflows resume --team-id <teamId> --run-id <runId>
```

Optionally, auto-resume any runs that already have decisions recorded:

```bash
openclaw recipes workflows poll-approvals --team-id <teamId>
```

## Debugging

- Inspect the run folder: `shared-context/workflow-runs/<runId>/`
- Check `run.json` events + node states
- Check `node-outputs/` for structured output
- Check `artifacts/` for tool output/files


## Outbound posting (side effects)

For side-effecting publish actions (e.g. posting to X/Discord), workflows should call an external Outbound Posting Service via the runner-native tool `outbound.post`.

See: `docs/OUTBOUND_POSTING.md`.
